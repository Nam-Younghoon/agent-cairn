# /discuss — 요구사항 논의 → 문서화 → 브랜치 생성

입력: $ARGUMENTS

이 커맨드는 이어지는 `/plan` → `/execute` → `/ship` 파이프라인의 **출발점**입니다.
산출물은 `__docs/` 디렉토리에만 저장되며, 실제 구현은 여기서 시작하지 않습니다.

## 서브에이전트 호출 흐름 (요약)
- **`parallel-explorer`**: 기존 구조·유사 기능 위치를 병렬로 훑을 때 (4단계 문서 작성 직전).
- `tdd-tester`, `pre-commit-reviewer` 는 이 커맨드에서 호출하지 **않는다**. 이후 단계에서 사용.

---

## 1단계. 대화형 요구사항 분석

사용자 설명을 읽고 아래를 **간결한 한국어**로 물어본다. 이미 답이 있으면 스킵한다.
- 문제 발생 조건/재현 경로 (버그인 경우)
- 신규 기능의 사용자 시나리오와 완료 기준
- 영향 범위: backend / web / mobile 중 어디?
- 스코프 밖으로 둘 항목
- UI 변경 필요 여부

최소 왕복으로 논의. 추측할 수 있는 부분은 가정으로 정리해 사용자에게 확인.

## 2단계. 산출물 제안

아래 형식 요약본을 사용자에게 보여주고 **"이대로 진행할까요?"** 컨펌을 받는다.

```
## 유형
feat | fix | chore | refactor | design | docs | test

## 제목 (브랜치명용, kebab-case)
예: login-error-500

## 배경 / 문제
...

## 요구사항 / 완료 기준
- [ ] ...

## 영향 범위
- [ ] backend
- [ ] web
- [ ] mobile

## 생성할 문서
- [x] PRD.md
- [ ] ARCHITECTURE.md
- [ ] ADR.md
- [ ] UI_GUIDE.md
```

## 3단계. 브랜치 생성

사용자 승인 후:
1. `git fetch origin`
2. `git switch dev && git pull --ff-only origin dev`
3. `git switch -c <type>/<kebab-요약>`
4. 생성된 브랜치명을 사용자에게 출력.

> `dev` 브랜치가 없으면 사용자에게 기준 브랜치를 물어본다.

## 4단계. (필요 시) 병렬 탐색

ARCHITECTURE.md 또는 PRD 의 "기존 구현" 항목을 채우기 위해 정보가 필요하면 **`parallel-explorer` 서브에이전트를 병렬로** 호출한다. 예:
- "기존 로그인 플로우가 어디에 있나"
- "Express 프로젝트의 에러 응답 규약은 무엇인가"
- "이 기능이 들어갈 feature 폴더 위치는 어디인가"

각 호출은 **독립**이어야 하며, 결과를 통합해 문서에 반영한다.

## 5단계. 문서 생성

`__docs/` 디렉토리를 만들고 아래 문서를 작성. **반드시** `.gitignore` 에 `__docs/` 가 포함되어 있는지 확인, 없으면 추가.

- `__docs/PRD.md` (필수)
- `__docs/ARCHITECTURE.md` (구조 변경이 있거나 이해에 필요한 경우)
- `__docs/ADR.md` (비자명한 의사결정이 있는 경우)
- `__docs/UI_GUIDE.md` (UI 변경이 있는 경우)

각 템플릿은 `.claude/templates/__docs/<name>.md` 에서 Read 한 뒤, `__docs/<name>.md` 로 복사·채워넣는다. **작성 지침(HTML 주석)을 빈 상태로 남겨두지 않는다**. 모르는 값은 사용자에게 묻는다.

## 6단계. 최종 승인 요청

문서 초안을 작성한 뒤 사용자 리뷰·수정을 요청. 승인이 있을 때까지 `/plan` 을 자동으로 이어가지 않는다.

### 다음 단계 안내
사용자에게 다음을 안내한다.

```
문서 초안을 만들었습니다. 리뷰 후 수정 요청을 주세요.
승인되면 /plan 으로 스텝을 분해할 수 있습니다.
```

---

## 금지
- 사용자 승인 전에 실제 코드 수정을 시작하지 않는다.
- `__docs/` 를 `.gitignore` 에 포함하지 않은 채로 파일을 커밋하지 않는다.
- 브랜치 생성 전에 `dev` 의 최신화 없이 분기하지 않는다.
- `PRD.md` 의 완료 기준을 모호한 문장(예: "잘 되도록 한다") 으로 남기지 않는다.
