# /discuss — 요구사항 논의 → 문서화 → 브랜치 생성

이 프롬프트는 Codex CLI 에서 `/discuss <설명>` 호출 시 Claude Code 의 동명 슬래시 커맨드와 **동일한 흐름**으로 동작합니다. Codex 에서는 커스텀 슬래시 커맨드 자동 바인딩이 확정되지 않았을 수 있으므로, 바인딩되지 않으면 `@.codex/prompts/discuss.md 의 절차를 따라 진행해주세요` 같이 파일 참조로 사용하세요.

입력: `<버그 또는 신규 기능 요약>`

이 커맨드는 이어지는 `/plan` → `/execute` → `/ship` 파이프라인의 **출발점**입니다.
산출물은 `__docs/` 디렉토리에만 저장되며, 실제 구현은 여기서 시작하지 않습니다.

---

## 1단계. 대화형 요구사항 분석

사용자 설명을 읽고 아래를 **간결한 한국어**로 물어본다. 이미 답이 있으면 스킵한다.
- 문제 발생 조건/재현 경로 (버그인 경우)
- 신규 기능의 사용자 시나리오와 완료 기준
- 영향 범위: backend / web / mobile 중 어디?
- 스코프 밖으로 둘 항목
- UI 변경 필요 여부

최소 왕복으로 논의. 추측할 수 있는 부분은 가정으로 정리해 사용자에게 확인.

## 2단계. 산출물 제안

아래 형식 요약본을 사용자에게 보여주고 **"이대로 진행할까요?"** 컨펌을 받는다.

```
## 유형
feat | fix | chore | refactor | design | docs | test

## 제목 (브랜치명용, kebab-case)
예: login-error-500

## 배경 / 문제
...

## 요구사항 / 완료 기준
- [ ] ...

## 영향 범위
- [ ] backend
- [ ] web
- [ ] mobile

## 생성할 문서
- [x] PRD.md
- [ ] ARCHITECTURE.md
- [ ] ADR.md
- [ ] UI_GUIDE.md
```

## 3단계. 브랜치 생성

사용자 승인 후:
1. `git fetch origin`
2. `git switch dev && git pull --ff-only origin dev`
3. `git switch -c <type>/<kebab-요약>`
4. 생성된 브랜치명을 사용자에게 출력.

> `dev` 브랜치가 없으면 사용자에게 기준 브랜치를 물어본다.

## 4단계. (필요 시) 병렬 탐색

ARCHITECTURE.md 또는 PRD 의 "기존 구현" 항목을 채우기 위해 정보가 필요하면 **아래 `## 인라인 가이드 — 병렬 탐색` 절차를 직접 따라** 탐색한다. Claude 의 `parallel-explorer` 서브에이전트와 동일한 역할을 Codex 세션 내에서 직접 수행한다.

각 질문은 독립적으로 다루고 결과를 통합해 문서에 반영한다. 예:
- "기존 로그인 플로우가 어디에 있나"
- "Express 프로젝트의 에러 응답 규약은 무엇인가"
- "이 기능이 들어갈 feature 폴더 위치는 어디인가"

## 5단계. 문서 생성

`__docs/` 디렉토리를 만들고 아래 문서를 작성. **반드시** `.gitignore` 에 `__docs/` 가 포함되어 있는지 확인, 없으면 추가.

- `__docs/PRD.md` (필수)
- `__docs/ARCHITECTURE.md` (구조 변경이 있거나 이해에 필요한 경우)
- `__docs/ADR.md` (비자명한 의사결정이 있는 경우)
- `__docs/UI_GUIDE.md` (UI 변경이 있는 경우)

각 템플릿은 `.codex/templates/__docs/<name>.md` (또는 `.claude/templates/__docs/<name>.md` 가 병설되어 있으면 그 경로) 에서 Read 한 뒤, `__docs/<name>.md` 로 복사·채워넣는다. **작성 지침(HTML 주석)을 빈 상태로 남겨두지 않는다**. 모르는 값은 사용자에게 묻는다.

## 6단계. 최종 승인 요청

문서 초안을 작성한 뒤 사용자 리뷰·수정을 요청. 승인이 있을 때까지 `/plan` 을 자동으로 이어가지 않는다.

### 다음 단계 안내
사용자에게 다음을 안내한다.

```
문서 초안을 만들었습니다. 리뷰 후 수정 요청을 주세요.
승인되면 /plan 으로 스텝을 분해할 수 있습니다.
```

---

## 금지
- 사용자 승인 전에 실제 코드 수정을 시작하지 않는다.
- `__docs/` 를 `.gitignore` 에 포함하지 않은 채로 파일을 커밋하지 않는다.
- 브랜치 생성 전에 `dev` 의 최신화 없이 분기하지 않는다.
- `PRD.md` 의 완료 기준을 모호한 문장(예: "잘 되도록 한다") 으로 남기지 않는다.

---

## 인라인 가이드 — 병렬 탐색

Claude 의 `parallel-explorer` 에이전트를 Codex 세션에서 직접 대체하는 절차. 탐색이 필요할 때 아래를 순서대로 따른다. **추측·제안·리팩터링 의견은 내지 않고 사실만 수집**.

### 작업 범위
- 특정 심볼/함수/파일의 위치와 사용처
- 특정 기능이 현재 어떻게 구현되어 있는지
- 설정 파일(`package.json`, `pubspec.yaml`, `tsconfig.json` 등) 현황
- 기존 아키텍처 패턴이나 레이어 구분 상태

### 절차
1. 질문을 분해해 **독립적인 검색 쿼리 목록**을 만든다.
2. `glob`/`grep`/`rg`/`fd` 를 가능한 한 **병렬로 실행**한다. 한 파일을 통째로 읽기 전에 패턴 검색을 먼저 한다.
3. **확인된 사실만** 기록한다. 파일 경로는 `path:line` 형식.

### 출력 포맷 (탐색 결과 보고 시 이 형식을 유지)
```
## 핵심 답
<2~3문장 요약>

## 근거
- <path>:<line> — <인용 또는 요약>
- ...

## 추가 탐색 제안 (선택)
- <필요하면 후속 질문만 나열, 답은 내지 않음>
```

### 금지
- 탐색 단계에서 코드 수정·파일 생성
- 탐색 진행 중 사용자에게 질문 (불명확한 부분은 탐색을 마친 뒤 메인 흐름에서 처리)
- 장황한 배경 설명
- 추측·의견 혼입 (탐색과 의사결정 단계를 분리)
